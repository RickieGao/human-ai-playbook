# 上下文工程

> "上下文就是基础设施" -- 环境结构不好，结果就差；设计好了，性能会复利式增长。

上下文工程是管理 AI "看到什么"的系统方法。它比"怎么问"（提示工程）更重要。

---

## 上下文的层次

```
Layer 1: CLAUDE.md          持久化的项目知识（最重要）
Layer 2: 项目结构            文件夹/文件命名本身就是上下文
Layer 3: Memory             跨 session 的记忆
Layer 4: 当前对话            当前 session 的上下文窗口
Layer 5: 工具输出            grep/find/read 的结果
```

## CLAUDE.md 设计原则

### 应该写什么
- AI 无法从代码推断的信息（特殊构建命令、项目特有约定）
- 踩过的坑和对应的规则
- 项目身份信息（技术栈、架构决策、协作模式）

### 不应该写什么
- AI 能从代码自行推断的信息（代码结构、函数签名）
- 过时的信息（比没有信息更糟）
- 过于冗长的内容（控制在 200 行以内）

### 关键研究发现
ETH Zurich 研究表明：
- LLM 自动生成的上下文文件降低了 3% 的成功率，且增加 20%+ 成本
- 人工编写的上下文文件带来 4% 的边际提升
- 结论：上下文文件必须由人编写，且限制在"AI 无法自行推断的信息"上

## 上下文窗口管理

200K token 的上下文窗口会快速填满。几乎所有最佳实践都围绕这个约束展开。

### 策略
1. **主动压缩**: 超过 50% 时手动 `/compact`
2. **精简工具**: 去掉不用的 MCP 工具（可能占用 8-30% 上下文）
3. **分 session**: 大任务拆分为多个 session，每个 session 有明确目标
4. **干净状态**: 每次 session 结束后留下可合并的代码、有 commit、有进度总结

## Memory 策略

### 应记住的
- 协作偏好（输出风格、审查严格度）
- 核心约定（命名规范、架构决策）
- 反复出现的指令
- 踩坑记录

### 不应记住的
- 临时调试信息
- 一次性上下文
- 可能快速过时的信息

## 项目结构即上下文

> 文件夹和文件结构本身就是一种上下文工程。

好的项目结构让 AI 仅通过文件名和目录层级就能理解项目意图。
命名要自解释，层级要有逻辑。
