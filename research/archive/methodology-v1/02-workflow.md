# 协作工作流框架

## 核心循环

```
Need -> Plan -> Execute -> Verify -> Document
  ^                                      |
  └──────────────────────────────────────┘
```

每个阶段都有明确的人机分工和交接标准。

---

## Phase 1: Need (需求澄清)

**主导者**: 人

- 用自然语言描述要做什么、完成标准是什么
- 如果需求复杂，让 AI 帮你整理和提问（但决策权在人）
- 输出：清晰的需求描述 + 验收标准

**检查点**: 你能否用一句话说清楚"要做什么"？如果不能，还没准备好进入下一阶段。

## Phase 2: Plan (规划)

**主导者**: 人机协作

- AI 探索代码库，理解现有结构（Explore -> Plan）
- AI 提出实现方案，人审查和调整
- 大任务拆分为可独立验证的小步骤
- 输出：分步实现计划

**检查点**: 每一步是否足够小、可验证、可回滚？

## Phase 3: Execute (执行)

**主导者**: AI（人监督）

- AI 按计划逐步实现
- 每完成一步，commit 一次
- 遇到计划外的问题，停下来和人讨论
- 输出：可运行的代码 + 对应的 commits

**检查点**: 代码是否可编译/运行？是否偏离了计划？

## Phase 4: Verify (验证)

**主导者**: 人机协作

- 运行测试（单元测试、集成测试）
- Code review（人审查 AI 产出）
- 检查是否满足需求描述中的验收标准
- 输出：验证通过 / 问题清单

**检查点**: 所有测试通过？人工审查无明显问题？

## Phase 5: Document (沉淀)

**主导者**: 人机协作

- 更新 CLAUDE.md（新的踩坑记录、约定）
- 更新相关文档
- 如果发现了可复用的模式，提取为 slash command 或规范
- 输出：更新后的协作规范

---

## 工作流变体

### 小任务（< 30 分钟）
直接 Need -> Execute -> Verify。不需要正式的 Plan 阶段。

### 大任务（> 2 小时）
严格执行完整循环。Plan 阶段可能需要多轮对话。考虑用多个 worktree 并行。

### Bug 修复
Need（复现 bug）-> Plan（定位根因）-> Execute（修复）-> Verify（确认修复 + 回归测试）-> Document（记录踩坑）
