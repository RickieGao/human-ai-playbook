# Kickoff Agent 重新设计

## 背景与问题

### 核心矛盾
用户需要 AI 帮忙生成 CLAUDE.md，但纯 AI 生成会导致：用户缺乏思考 → 内容浮于表面 → 没有 ownership → 不维护 → 失效。

### 现有流程的问题（按严重程度排序）
1. **缺少代码分析** — 能自动检测的信息（技术栈、命令）还在问用户，浪费时间
2. **问题太抽象** — 如"AI 自主级别"，用户没有直觉，选了等于没选
3. **问题太多** — 3 轮 17 个问题，用户疲劳，到后面随便填
4. **缺少教育** — 用户不理解为什么这些选择重要，导致对生成内容没有 ownership

### 目标用户
分层覆盖所有用户：
- **新手** — 连 CLAUDE.md 是什么都不太清楚
- **中级** — 用过但没写过 CLAUDE.md，或写了但效果不好
- **有经验** — 理解 CLAUDE.md 的价值，但觉得从零写太费时间

## 定位

**Kickoff Agent 是用户有创造想法时，与 AI 沟通的第一步。** 引导用户从模糊念头走向清晰方案。最终结果是找到一个合适的、满足需求的方案，或者创造一个新的。

不只是 CLAUDE.md 生成器，而是**项目启动的第一道关卡**。

## 方案选择

评估了三个方案：

1. **代码先行** — AI 分析 + 用户补盲区。效率高但用户 ownership 弱。
2. **教练模式** — 引导式对话 + 场景化选择。ownership 强但对老手啰嗦。
3. **智能混合（选定）** — 融合 1 和 2：自动检测客观事实 + 场景化引导主观偏好 + 分层体验 + 审查即教育。

选定方案 3，因为它同时解决了四个痛点：自动检测解决"缺少代码分析"，场景化解决"太抽象"，分层解决"太多"，审查即教育解决"缺少教育"。

## 整体流程

```
感知环境
  ├── 有项目文件 → 路径 A：项目扫描 → 阶段 2
  └── 空目录/无项目 → 路径 B：从想法开始
                        ├── Step 1: 需求澄清（头脑风暴）
                        ├── Step 2: 现有方案调研
                        │   ├── 已有方案满足 → 推荐，结束 ✓
                        │   ├── 部分满足 → 讨论二开 vs 新建
                        │   └── 不满足 → Step 3: 技术方案建议
                        └── 合流 ↓
阶段 2：场景化引导（协作偏好）
阶段 3：CLAUDE.md 初稿生成
阶段 4：逐段审查（教育 + ownership）
阶段 5（可选）：项目脚手架初始化
```

### 核心原则
- 能自动检测的不问用户
- 能用场景描述的不用抽象概念
- 每一步用户都有选择权和修改权
- 帮用户想清楚"要不要做"比"怎么做"更重要

---

## 路径 B：从想法到项目

### Step 1 — 需求澄清（头脑风暴模式）

整个流程中**最重要的阶段**，不设固定问题数量，以对话质量为准。

**进入方式**：AI 检测到空目录/无项目文件，用户描述了想法后自动进入。

**目标**：产出一份清晰的需求摘要，包含：
- **核心问题**：解决什么痛点
- **目标用户**：谁用、什么场景
- **MVP 边界**：做什么、明确不做什么
- **成功标准**：怎么判断做成了

**AI 的角色是引导者，不是记录员。** 具体行为：
1. **追问本质** — 用户说"想做 X"，AI 问"为什么需要 X？现在怎么解决这个问题的？"
2. **挑战假设** — "你提到需要多用户协作，但如果先只做单人版，能解决核心问题吗？"
3. **收窄范围** — 每当用户添加新功能，AI 帮助判断"这是 MVP 必需的，还是后续迭代的？"
4. **具象化** — "你能描述一个典型的使用场景吗？从打开到完成，你期望的步骤是什么？"
5. **总结确认** — 讨论到一定深度后，AI 输出需求摘要让用户确认

**退出条件**：当核心问题、目标用户、MVP 边界、成功标准都清晰且用户确认后，进入市场调研。

**示例对话**：

> 用户：我想做一个记账工具
>
> AI：记账工具有很多种。你现在是怎么记账的？遇到了什么不满意的地方？
>
> 用户：用 Excel 记的，太麻烦了，经常忘记
>
> AI：所以核心痛点是"记录太麻烦导致坚持不下来"。那你觉得理想的记账体验是什么样的？比如：每次花钱后 10 秒内完成记录？还是每天晚上花 5 分钟回顾？
>
> 用户：最好能自动，不用手动记
>
> AI：自动记账通常意味着对接银行/支付 API，这在技术和合规上都很复杂。我们可以讨论几个折中方案：a) 拍照小票自动识别 b) 微信/支付宝账单导入 c) 语音快速记一笔。你觉得哪个方向更实际？

### Step 2 — 现有方案调研

**进入条件**：需求摘要已确认。

AI 基于需求摘要搜索和评估已有方案，结构化呈现：

| 方案 | 匹配度 | 优势 | 缺什么 |
|------|--------|------|--------|
| 随手记 | 高 | 成熟、分类统计完善 | 不支持语音记账 |
| Beancount | 中 | 开源、可定制 | 需要技术基础 |

三种走向：
- **推荐现有方案** → 流程结束（帮用户省了时间）
- **基于开源二开** → 进入技术方案（已有代码库基础）
- **从零新建** → 进入技术方案

**关键原则**：AI 应该诚实推荐最优解，即使最优解是"不需要自己做"。这建立用户信任。

### Step 3 — 技术方案建议

**进入条件**：调研后确认需要自己做。

1. **评估用户技术水平** — 不直接问"你会什么语言"，而是自然了解：
   > "你之前有没有做过类似的项目？或者平时接触过哪些编程语言/工具？完全没经验也没关系。"

2. **推荐方案** — 针对用户水平：
   - 零基础：推荐最简单、社区资源最多、AI 辅助效果好的方案
   - 有经验：给出 2 个方案对比

3. **确认后输出技术摘要**：技术栈选型及理由、项目大致结构、推荐的开发工具

**关键原则**：
- 零基础用户选型时优先考虑"AI 能辅助的程度"
- 不堆砌选项，避免选择焦虑
- 方案与 MVP 边界匹配，不过度设计

## 路径 A：项目扫描

**进入条件**：检测到当前目录有项目文件。

### 自动检测项

| 检测来源 | 提取信息 |
|----------|----------|
| 包管理文件（package.json 等）| 语言、框架、依赖 |
| 目录结构 | 项目类型（前端/后端/全栈/monorepo）|
| scripts / Makefile | build、test、lint 命令 |
| CI 配置（.github/workflows 等）| CI/CD 流程 |
| 代码规范文件（.eslintrc、.prettierrc 等）| 编码约定 |
| git log | 提交风格、contributor 数量、活跃度 |
| 已有 CLAUDE.md | 是否是更新而非首次创建 |

### 输出

一份扫描报告，用户确认或修正：

> "我扫描了你的项目，检测到：
> - **技术栈**：TypeScript + React 18 + Vite，pnpm 管理
> - **测试**：Vitest（`pnpm test`）
> - **代码规范**：ESLint + Prettier，已有配置
> - **CI**：GitHub Actions，push 时运行测试和 lint
> - **团队**：最近 30 天 3 位 contributor
> - **提交风格**：conventional commits（feat/fix/docs）
>
> 以上正确吗？有需要补充或修正的？比如我可能漏掉的工具、特殊约定、或已知的坑？"

最后一句主动邀请用户补充 AI 扫描不出来的隐性知识。

## 阶段 2：场景化引导

**进入条件**：路径 A 扫描确认完毕，或路径 B 技术方案确认完毕。

### 模式选择

> "接下来我想了解你希望怎么和 AI 协作。有两种方式：
> - **快速模式**（3 个问题，约 2 分钟）— 覆盖最核心的协作偏好
> - **完整模式**（7 个问题，约 5 分钟）— 更细致地定制协作方式
>
> 选哪个？"

路径 B 零基础用户默认进入完整模式，不提供选择。

### 快速模式（3 个场景问题）

**Q1 — 工作节奏**
> "你让 AI 帮你实现一个用户登录功能。你希望 AI 怎么做？
> a) 先写个方案给你看，你同意了再动手写代码
> b) 直接写代码，但每完成一小步就给你看
> c) 一口气把整个功能做完，你最后统一看结果"

→ 映射：工作流（plan-first / incremental / high-autonomy）

**Q2 — 意外处理**
> "AI 在实现你要的功能时，发现有个已有的函数写得不好，改了会让整体更干净。AI 应该？
> a) 只做你要求的，别的不碰
> b) 告诉你发现了问题，问你要不要一起改
> c) 顺手改了，在结果里告诉你改了什么"

→ 映射：自主级别 + 变更范围控制

**Q3 — 沟通方式**
> "AI 完成一个任务后，你希望它怎么汇报？
> a) 详细解释做了什么、为什么这么做、还有哪些替代方案
> b) 简要说明改了什么、关键决策是什么
> c) 直接给代码差异，不用多解释"

→ 映射：沟通风格

### 完整模式（额外 4 个问题）

**Q4 — 测试态度**
> "关于测试，你的期望是？
> a) 每写一个功能必须有对应的测试
> b) 核心逻辑要有测试，边缘情况看情况
> c) 先把功能做出来，测试以后再补"

→ 映射：质量标准 / 测试要求

**Q5 — 代码审查**
> "你想多频繁地检查 AI 的工作？
> a) 每改一个文件我都要看
> b) 完成一个完整功能后我来审查
> c) 我信任 AI，提交 PR 时统一看"

→ 映射：审查频率

**Q6 — 已知坑点**
> "这个项目里有没有什么'坑'是 AI 容易踩的？比如：某个 API 有隐藏的限制、某个目录不能动、某个依赖有已知 bug？（没有的话跳过）"

→ 映射：已知坑点 / 注意事项

**Q7 — 安全红线**
> "有没有什么 AI 绝对不能做的事？比如：不能删除数据库、不能访问生产环境、不能修改某些配置文件？（没有的话跳过）"

→ 映射：安全约束 / 禁止操作

### 总结确认

选完后 AI 总结协作偏好，用户确认后进入下一阶段。

## 阶段 3：CLAUDE.md 初稿生成

**进入条件**：场景化引导完成，用户确认协作偏好。

### 数据来源

| 内容 | 路径 A 来源 | 路径 B 来源 |
|------|------------|------------|
| 项目名/描述 | 扫描检测 | 需求摘要 |
| 技术栈 | 扫描检测 | 技术方案建议 |
| 构建/测试命令 | 扫描检测 | 技术方案推断 |
| 代码规范 | 扫描检测 | 技术方案默认值 |
| 工作流 | Q1 映射 | Q1 映射 |
| 自主级别 | Q2 映射 | Q2 映射 |
| 沟通风格 | Q3 映射 | Q3 映射 |
| 坑点/安全红线 | Q6/Q7 + 扫描补充 | Q6/Q7 |

模板体系保持不变：base/ + workflow/ + scale/ 的片段拼装。改变的是数据采集方式，不是模板本身。

生成后不直接写入文件，进入阶段 4。

## 阶段 4：逐段审查（教育 + ownership）

**进入条件**：初稿生成完毕。

**核心理念**：解决"AI 生成 → 用户没 ownership → 不维护"问题的关键环节。不是把整个文件丢给用户，而是逐段引导审查。

### CLAUDE.md 拆分为 4-5 段展示

**第 1 段：项目身份**
展示项目名、描述、技术栈、常用命令。
> **这段的作用**：AI 每次开始工作时首先读取这里，确保它了解项目是什么、怎么运行。

**第 2 段：协作模式**
展示工作流、变更范围、沟通风格、审查频率。
> **这段的作用**：这定义了你和 AI 的"工作合同"。AI 会严格按这些规则行事。

**第 3 段：质量标准**
展示测试要求、提交规范、代码风格。
> **这段的作用**：防止 AI 产出质量不一致的代码。

**第 4 段：注意事项和红线**
展示已知坑点、安全约束、禁止操作。
> **这段的作用**：这是"防护栏"。最重要的一段，因为内容是 AI 从代码中看不出来的。
> 路径 B 用户额外提示："这段现在可能比较少，随着项目推进你会发现越来越多的坑，记得回来补充。"

**第 5 段（如有）：上下文管理**
展示 CLAUDE.md 维护规则、会话管理。

### 每段交互模式
1. 展示内容
2. 一句话解释作用（路径 B 用户附加具体例子）
3. 等待用户确认/修改/补充
4. 用户确认后进入下一段

全部确认后写入文件。

## CLAUDE.md 自我更新机制

### 核心原则
AI 可以发现更新需求、起草更新内容，但**每次更新必须经用户审核同意**。

### 触发时机

| 触发信号 | 更新建议 |
|----------|----------|
| 新增了依赖/工具 | 补充技术栈和相关命令 |
| 用户多次纠正同一个行为 | 将纠正固化为规则 |
| 发现了新的坑 | 补充到注意事项 |
| 用户改变了协作偏好 | 更新协作模式 |
| 项目结构发生重大变化 | 更新项目身份和规模配置 |
| CLAUDE.md 中的命令已失效 | 修正命令 |

### 更新流程

```
AI 检测到更新信号
  → 在合适的时机提出（不打断当前工作）
    → 展示：当前内容 vs 建议修改 + 修改原因
      → 用户同意 → 写入
      → 用户拒绝 → 不改，记住用户的判断
      → 用户修改 → 按用户版本写入
```

**示例**：
> "我注意到你最近三次都让我不要自动加注释。建议在 CLAUDE.md 中加一条规则：
> ```diff
> ## 协作模式
> + - 不要在代码中添加解释性注释，除非我明确要求
> ```
> 要加上吗？"

### 行为约束
- **不打断当前工作** — 在任务完成后或自然停顿时提出
- **批量建议** — 多个小更新合并为一次建议
- **不重复建议** — 用户拒绝过的不再提出，除非情况明显变化
- **保持精简** — CLAUDE.md 保持在 200 行以内

### 职责归属
- **Kickoff Agent** — 初始生成 + 提供更新机制框架
- **日常协作中的 AI** — 执行检测和建议（任何会话都可触发）
- **Retrospective Agent（v2）** — 周期性回顾，深层优化建议

## 与现有设计的变更总结

| 维度 | 现有设计 | 新设计 |
|------|---------|--------|
| 定位 | CLAUDE.md 生成器 | 项目启动第一道关卡 |
| 路径 B | 无 | 需求澄清 → 调研 → 技术方案 |
| 数据采集 | 17 问访谈 | 自动扫描 + 3-7 场景问题 |
| 问题风格 | 抽象概念 | 具体场景 |
| 输出审查 | 整体展示 | 逐段审查 + 教育 |
| 维护 | 无 | 自我更新机制 |
| 用户分层 | 无 | 快速/完整模式 |
| 脚手架 | 无 | 可选 |

## 需要保留的现有设计

- **模板片段体系**（base/ + workflow/ + scale/）— 拼装逻辑不变
- **Meta Agent 通信原则** — 所有交互遵循
- **200 行上限** — 保持 CLAUDE.md 精简
- **决策树逻辑** — 场景问题的答案仍映射到相同的模板选择
