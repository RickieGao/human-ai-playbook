## 一、Anthropic 官方信源

### 1. 《Claude Code: Best Practices for Agentic Coding》
- **平台**：Anthropic Engineering Blog
- **作者**：Boris Cherny（Claude Code 创始人/负责人）
- **链接**：https://www.anthropic.com/engineering/claude-code-best-practices
- **核心观点**：
  - 核心工作流为「Explore → Plan → Code → Commit」，先探索后规划是关键
  - CLAUDE.md 是 AI 的持久记忆，应记录项目约定、命令、踩坑记录
  - TDD（测试驱动开发）在 AI 协作中变得更强大：先写测试→确认失败→再写实现
  - 用 git worktree 实现多个 Claude 实例并行工作
  - 上下文管理是第一优先级，超过 50% 时手动 /compact

### 2. 《Effective Harnesses for Long-Running Agents》
- **平台**：Anthropic Engineering Blog
- **作者**：Justin Young 等 Anthropic 工程团队
- **链接**：https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents
- **核心观点**：
  - 长时运行 agent 的关键是「初始化 agent + 编码 agent」两阶段架构
  - 用 claude-progress.txt 文件记录进度，让新 context window 快速了解工作状态
  - 增量推进：每次只做一个 feature，避免 agent 一次做太多
  - 每次 session 结束后留下"干净状态"：代码可合并、有 commit、有进度总结

### 3. 《Building Agents with the Claude Agent SDK》
- **平台**：Anthropic Engineering Blog
- **链接**：https://www.anthropic.com/engineering/building-agents-with-the-claude-agent-sdk
- **核心观点**：
  - Agent 的核心循环是「收集上下文 → 采取行动 → 验证工作 → 重复」
  - 文件夹和文件结构本身就是一种上下文工程
  - 优先使用 agentic search（grep/find），而非语义搜索（RAG）
  - 子 agent 是天然支持的，可按职责分工

### 4. 《Scaling Agentic Coding Across Your Organization》
- **平台**：Anthropic Resources
- **链接**：https://resources.anthropic.com/scaling-agentic-coding
- **核心观点**：
  - 从少数 power user 到全组织推广的实践手册
  - 安全优先：通过 CLAUDE.md 和 /security-review 命令落地
  - 避免"什么都让 AI 做"的陷阱，TDD 是验证 AI 输出的最佳手段
  - 衡量 ROI 要超越虚荣指标

### 5. 《2026 Agentic Coding Trends Report》
- **平台**：Anthropic Resources (PDF)
- **链接**：https://resources.anthropic.com/hubfs/2026%20Agentic%20Coding%20Trends%20Report.pdf
- **核心观点**：
  - 2025 年 agentic AI 改变了大量开发者写代码的方式，2026 年将重塑 SDLC
  - 工程师描述对"容易验证的任务"委托给 AI，概念性或设计性强的任务保留给自己或协作完成
  - 多 agent 编排是趋势：单 agent 顺序处理 → 多 agent 并行 + 综合

## 五、学术论文与研究报告

### 1. METR — 《Measuring the Impact of Early-2025 AI on Developer Productivity》
- **平台**：METR.org（非营利研究机构）
- **作者**：Joel Becker, Nate Rush, Elizabeth Barnes, David Rein
- **链接**：https://metr.org/blog/2025-07-10-early-2025-ai-experienced-os-dev-study/
- **核心发现**：
  - 随机对照试验：资深开发者使用 AI 工具后**慢了 19%**
  - 开发者自我评估认为快了 20%——存在 39 个百分点的认知偏差
  - 可能原因：对 AI 过度乐观、在熟悉项目上 AI 帮助有限、大型仓库的隐性规则 AI 不理解
  - **关键启示：人机协作需要刻意训练，不是"用了就快"**
- **后续更新**（2026.02）：https://metr.org/blog/2026-02-24-uplift-update/
  - 越来越多开发者拒绝在无 AI 条件下工作，导致研究设计面临选择偏差

### 2. ETH Zurich — AGENTS.md 效果研究
- **平台**：arXiv + InfoQ 报道
- **作者**：Thibaud Gloaguen, Niels Mündler, Mark Müller 等
- **链接**：
  - 论文：https://arxiv.org/abs/2510.21413
  - InfoQ 报道：https://www.infoq.com/news/2026/03/agents-context-file-value-review/
- **核心发现**：
  - LLM 自动生成的上下文文件**降低了** 3% 的成功率，且增加 20%+ 推理成本
  - 人工编写的上下文文件带来 4% 的边际提升，但也增加成本
  - **关键启示：上下文文件应由人编写，且限制在"AI 无法自行推断的信息"上**（如特殊构建命令、项目特有约定）

### 3. Anthropic — 《2026 Agentic Coding Trends Report》
- **平台**：Anthropic Resources
- **核心发现**：
  - 工程师对"什么该委托给 AI"逐渐形成直觉：容易验证的 → 委托；概念性/设计性强的 → 自己做或协作
  - 多 agent 编排：Writer/Reviewer 模式、Plan/Execute 分离

## 九、跨信源的共识性结论

综合以上所有信源，以下原则在多个独立来源中反复出现：

1. **上下文工程 > 提示工程**：管理好 AI "看到什么"比"怎么问"更重要
2. **先规划后执行**（Plan → Execute）：几乎所有成功案例都强调这一点
3. **增量推进 + 频繁 commit**：小步快跑 + git 回滚能力
4. **CLAUDE.md 作为活文档**：记录踩坑、持续更新、由人编写而非 AI 生成
5. **验证闭环不可省略**：TDD、code review、安全审计
6. **人机分工清晰**：人做决策/架构/审查，AI 做实现/重复/测试
7. **渐进式信任**：从小任务开始，逐步扩大 AI 自主权
8. **多 agent 并行**是高级模式，但需要编排策略
9. **AI 不一定让你更快**：METR 研究证明认知偏差的存在，需要客观衡量
10. **知识沉淀为可复用资产**：slash commands, skills, hooks, 规范文档
